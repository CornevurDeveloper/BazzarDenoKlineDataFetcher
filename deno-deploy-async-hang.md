# Решение проблемы: "Зависание" (Suspend) фоновых задач на Deno Deploy

## Симптомы проблемы
При запуске долгих тяжелых задач (парсинг API, агрегация данных) через HTTP-запрос на Deno Deploy происходило следующее:
1. Запрос успешно стартовал, в логах отображался нормальный прогресс.
2. Спустя несколько секунд (когда обрабатывалось около 30-40 сущностей из очереди) логи **внезапно полностью останавливались**.
3. Не возникало никаких ошибок, исключений (`catch`) или сообщений об `Out of Memory`.
4. Спустя еще какое-то время Deno Deploy просто перезапускал инстанс с нуля (появлялись стартовые логи `[MONGO] Connected to DB`).
5. Локально (на домашнем ПК через `deno task start`) этот же код отрабатывал на 100% идеально без единой запинки.

## Причина: "Isolate Suspension" (Засыпание процесса)

Архитектура Serverless-систем, включая **Deno Deploy** (а также Cloudflare Workers, Vercel Edge Functions), устроена так, чтобы экономить процессорное время. 

Единицей исполнения является V8 Isolate (облегченный контекст). Платформа держит Isolate "пробужденным" (awake) и дает ему процессорное время **только пока обрабатывается хотя бы один активный сетевой (HTTP) запрос**.

В оригинальном `server.ts` эндпоинт запуска работы был написан по стандартным канонам классического Node.js:
```typescript
app.post("/api/jobs/run/:jobName", (req, res) => {
  const jobToRun = jobs[jobName];

  // 1. Функция запускается в "фоне"
  jobToRun(); 

  // 2. Сервер СРАЗУ отправляет ответ пользователю, закрывая HTTP запрос
  return res.status(202).json({ success: true }); 
});
```

### Что происходило под капотом:
1. Шлюз Deno Deploy получал запрос и будил процесс.
2. Фоновая задача `jobToRun()` начала выполняться в циклах.
3. Node.js (Express) моментально отдавал `202 Accepted` клиенту.
4. HTTP-запрос считался завершенным и соединение закрывалось.
5. Инфраструктура Deno Deploy видела, что активных подключений больше нет, поэтому для экономии CPU **принудительно усыпляла (suspend) V8 Isolate**. Все `fetch` запросы, `setTimeout` и интервалы тупо замораживались. Отсюда — внезапное молчаливое зависание на `30/78` монетах.
6. Позже, когда приходил другой запрос (`/health`), процесс просыпался на пару миллисекунд (зачастую с уже оборванными TCP-сокетами для fetch), и затем снова уходил в сон. Если процесс долго "висел" в фоне, Deno deploy его просто стирал как мусорный (garbage collect).

## Окончательное Решение

Чтобы Deno Deploy позволил процессу закончить работу, мы **обязаны удерживать HTTP-соединение открытым** до самого конца исполнения задачи. 

Однако, если мы просто напишем `await jobToRun()`, многие обратные прокси (Reverse Proxies, например Cloudflare/Nginx перед Deno) могут принудительно разорвать соединение по таймауту (Gateway Timeout), если сервер молчит больше 30-60 секунд.

**Лучший паттерн для долгих Serverless-задач — Streaming HTTP Response (Keep-Alive + Chunked)**:

```typescript
app.post("/api/jobs/run/:jobName", async (req, res) => {
  const jobToRun = jobs[jobName];
  
  // 1. Отвечаем браузеру чанками (кусками), держим связь открытой!
  res.writeHead(200, {
    "Content-Type": "application/json",
    "Transfer-Encoding": "chunked",
    "Connection": "keep-alive" // Принуждаем Isolate не спать
  });
  
  res.write(`{"status":"running","job":"${jobName}","progress":"`);
  
  // 2. Периодически "капаем" байтами, чтобы шлюзы (прокси) не отвалились по таймауту неактивности
  const interval = setInterval(() => res.write("."), 10000);

  try {
    // 3. ДОЖИДАЕМСЯ конца работы в рамках этого же открытого запроса
    await jobToRun(); 
    res.write(`","success":true}`);
  } catch (err: any) {
    const errMsg = err instanceof Error ? err.message : String(err);
    res.write(`","success":false,"error":${JSON.stringify(errMsg)}}`);
  } finally {
    // 4. Убиваем пингатор и КОНЕЧНО закрываем запрос
    clearInterval(interval);
    res.end(); 
  }
});
```

### Дополнительная страховка: `AbortSignal`
Помимо "засыпания", `fetch`-запросы иногда могут зависать из-за банального дропа TCP-пакетов (когда биржа не ответила, а сокет остался открыт вечно). В NodeJS/Deno нет дефолтного таймаута на `fetch`. 

**Всегда** нужно прописывать таймаут на внешний API, чтобы очередь (`Promise.all`) не блокировалась навечно:
```typescript
const response = await fetch(requestUrl, {
  headers: { "Accept": "application/json" },
  signal: AbortSignal.timeout(10000) // 10 секунд на жесткий таймаут
});
```

Внедрение этих двух изменений гарантирует 100% прохождение долгих заданий на платформе Deno Deploy.
